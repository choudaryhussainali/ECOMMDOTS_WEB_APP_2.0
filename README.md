### Phase 1: The Engine & Data Layer (Backend)

**1. `app.py` (The Central Nervous System)**
This is your Flask server. It listens to the web. When a user requests a URL, `app.py` intercepts it, connects to the database, fetches the exact mathematical amount of data needed, and passes it to the `templates/` folder to be rendered into HTML. It controls three primary routes:

* `@app.route('/')`: The homepage.
* `@app.route('/article/<slug>')`: The single-article generator.
* `@app.route('/api/load_more')`: The hidden asynchronous API for category filtering and pagination.

**2. `ecommdots.db` (The Memory Bank)**
Your lightweight, lightning-fast SQLite database. It contains an `articles` table with columns for `title`, `slug`, `category`, `excerpt`, `image_url`, and the raw HTML `content` payload generated by your editor.

---

### Phase 2: The UI Architecture (Frontend Templates)

**3. `base.html` (The Master Wrapper)**
This file enforces **D.R.Y. (Don't Repeat Yourself)** architecture. It contains your global `<head>` tags, your floating platinum `<header>`, your `<footer>`, and your elite custom cursor/kinetic JavaScript. Every other page in your project gets injected exactly into the middle of this file using the `{% block content %}` Jinja2 tag.

**4. `blog_grid.html` (The Root Homepage)**
Since we purged the agency pages, this is your front door. It contains the cinematic hero image and the sticky, glassmorphism category navigation. When it loads, it dynamically generates the category buttons based on the exact counts in your database.

**5. `_blog_loop.html` (The Injection Fragment)**
A masterful architectural trick. Instead of writing the HTML for your blog cards twice (once for the initial page load, and once for the JavaScript "Load More" function), we wrote it here once. Both the homepage and the API pull from this exact file to generate cards, ensuring zero code duplication.

**6. `article.html` (The Reading Layout)**
The destination. When a user clicks a card, this file catches the specific article data. It forces the screen into a blackout mode, applies an elite reading-width container, and utilizes the Jinja2 `| safe` filter to inject your TinyMCE payload as pure, flawlessly formatted HTML.

**7. `admin.html` (The Command Center)**
Your secure, hidden backend portal. It contains the TinyMCE WYSIWYG editor where you write your intelligence reports, format text, and assign categories before deploying them to the live database.

### Phase 3: The Complete CMS Workflow (How Data Moves)

To truly master this application, you must understand the exact lifecycle of an article. Here is the step-by-step pipeline from creation to the reader's screen.

#### Step 1: Ingestion (Creation)

1. **The Input:** You navigate to your hidden `/admin` portal. You type your intelligence into the TinyMCE editor, format it, and fill out the metadata (Title, Category, Excerpt, Image URL).
2. **The Processing:** You click "Deploy". `app.py` intercepts this form submission. It sanitizes the data, auto-generates a URL-friendly `slug` (e.g., `amazon-ppc-strategy`), and permanently writes the row into `ecommdots.db`.

#### Step 2: The Initial Render (First Load)

1. **The Request:** A user navigates to `ecommdots.com`.
2. **The Database Query:** `app.py` queries `ecommdots.db`: *"Give me the 10 most recent articles, and calculate the total count of articles in every category."*
3. **The Build:** `app.py` hands this data to `blog_grid.html`. The page uses `_blog_loop.html` to instantly draw the first 10 cards and populate the category buttons with the correct numbers.

#### Step 3: The Asynchronous Filter Engine (Kinetic Filtering)

1. **The Interaction:** The user clicks the **"PPC & Ads"** category button on the homepage.
2. **The Stealth Request:** The page *does not reload*. The JavaScript in `base.html` dims the grid and quietly fires a `fetch()` request to `app.py` at the `/api/load_more` endpoint.
3. **The API Response:** `app.py` asks the database for only PPC articles. It passes them through `_blog_loop.html` to generate a string of raw HTML cards. The API sends this HTML chunk (along with the new total count) back to the browser.
4. **The Paint:** The JavaScript instantly replaces the old grid with the new cards, recalculates the math for the "Load More" button, and elegantly fades the grid back in.

#### Step 4: The Delivery (Reading)

1. **The Click:** The user clicks on a specific article card.
2. **The Fetch:** The browser navigates to `/article/amazon-ppc-strategy`.
3. **The Unpacking:** `app.py` reads the URL, searches the database for that exact slug, and pulls the massive HTML text payload.
4. **The Experience:** It hands the payload to `article.html`, which wraps the text in your dark-mode editorial layout, presenting a million-dollar reading experience.
